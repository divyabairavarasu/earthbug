## 🔴 CRITICAL — App is broken in production

### BUG-01 · Gemini model `gemini-1.5-flash` returns HTTP 404
**File:** `src/utils/gemini.js:39`
**Test:** `analysis-flow.spec.js › CRITICAL-BUG: request URL uses gemini-1.5-flash which is 404 on v1beta`

The `GEMINI_MODEL` constant is set to `'gemini-1.5-flash'`. The SDK (v0.21.0) calls the `v1beta` endpoint. Google has removed this model from the `v1beta` API surface, returning:

```
404 models/gemini-1.5-flash is not found for API version v1beta,
or is not supported for generateContent
```

**Fix:** Change `GEMINI_MODEL` to a supported model (e.g. `'gemini-2.0-flash'` or `'gemini-1.5-flash-001'`). Verify against the [ListModels](https://generativelanguage.googleapis.com/v1beta/models) response for the account's tier.

## 🔴 HIGH — Misleading error messages

### BUG-02 · 404 "model not found" shows generic, unactionable error
**File:** `src/utils/gemini.js:78–100` (`formatGeminiError`)
**Test:** `error-handling.spec.js › CRITICAL-BUG: 404 (model not found) shows unhelpful generic error`

When the API returns 404, none of the error patterns match (no `429`, `401`, `403`, `api key`, or `network` in the message). The function falls through to the default:

> "EarthBug could not analyze that photo right now. Please try again."

Users have no idea the app is misconfigured. **Fix:** Add a 404 / `NOT_FOUND` pattern and show a message like: "The AI model is unavailable — please contact support."

### BUG-03 · `AUTH_FAILED_PATTERN = /401|403/` is too broad (false positives)
**File:** `src/utils/gemini.js:50`
**Test:** `error-handling.spec.js › KNOWN-BUG: AUTH_FAILED_PATTERN falsely matches strings containing "401"`

The regex `/401|403/` matches any string containing those digit sequences (e.g. `"batch 40138 failed"` → shown as "API key rejected"). **Fix:** Match the HTTP status code exactly, e.g. check `error.status` or use `/(^|\D)40[13](\D|$)/`.

### BUG-04 · `QUOTA_EXCEEDED_PATTERN = /429/` is too broad (false positives)
**File:** `src/utils/gemini.js:48`
**Test:** `error-handling.spec.js › KNOWN-BUG: QUOTA_EXCEEDED_PATTERN falsely matches strings containing "429"`

`/429/` matches `"batch 4299 failed"` → shown as quota error. **Fix:** Same as BUG-03 — anchor the status code to word boundaries or use structured error fields.

## 🟡 MEDIUM — UX / functional gaps

### BUG-05 · No cancel or timeout on hung Gemini request
**File:** `src/App.jsx:96–118` (`analyzeImage`), `src/utils/gemini.js:125`
**Test:** `error-handling.spec.js › KNOWN-BUG: hung Gemini request leaves user permanently on analyzing screen`

If the Gemini API never responds, the user is locked on the Analyzing screen with no escape. **Fix:** Wrap `identifyBug` in a `Promise.race` with a timeout (e.g. 30 s), and add a "Cancel" button to `AnalyzingView`.

### BUG-06 · File input value not cleared on upload error (re-select same file fails)
**File:** `src/components/CameraView.jsx:59–62`
**Test:** `file-upload.spec.js › KNOWN-BUG: re-selecting same file after error does not re-trigger upload`

On success, `fileInputRef.current.value = ''` is called. On error (image processing failure or non-image type), it is NOT called. Re-selecting the same file fires no `change` event — the user appears stuck. **Fix:** Call `fileInputRef.current.value = ''` in the `catch` block too.

### BUG-07 · GIF files bypass compression → potentially huge API payload
**File:** `src/utils/imageUtils.js:76`
**Test:** `file-upload.spec.js › KNOWN-BUG: GIF upload bypasses compression and sends raw base64`

`BYPASS_COMPRESSION_MIME_TYPES` skips GIF (and SVG) from the canvas compression step. A large animated GIF could send megabytes of base64 to Gemini. **Fix:** Convert GIF frames to JPEG before analysis, or reject GIFs with a file-size warning.

### BUG-08 · Scan history lost on page refresh (no localStorage persistence)
**File:** `src/App.jsx:38` (`useState([])`)
**Tests:** `scan-history.spec.js`, `persistence.spec.js › KNOWN-BUG: scan history is not persisted`

History is stored in React state only. Refreshing destroys it. **Fix:** Persist `scanHistory` to `localStorage` (storing thumbnail data URLs and analysis results), reading it back on mount.

### BUG-09 · Duplicate scan history entries (no deduplication)
**File:** `src/App.jsx:106–109`
**Test:** `scan-history.spec.js › KNOWN-BUG: scanning same bug multiple times adds duplicate history entries`

The same bug scanned multiple times creates duplicate thumbnails. **Fix:** Deduplicate by `name` (or a hash of the analysis result) before prepending.

## 🟡 MEDIUM — Accessibility

### BUG-10 · Missing `alt` attribute when `result.name` is undefined
**File:** `src/components/ResultsView.jsx:99`
**Test:** `results-view.spec.js › KNOWN-BUG: missing name in result renders img alt as "undefined"`

`<img alt={result.name} …>` — when Gemini returns partial JSON without a `name` field, `result.name` is `undefined`. React omits the attribute entirely (no `alt` at all). Screen readers announce the image URL instead. **Fix:** `alt={result.name ?? 'Identified bug'}`.

### BUG-11 · Two `role="status"` elements exist simultaneously
**File:** `src/components/ResultsView.jsx:114` and `src/components/ResultsView.jsx:154`
**Test:** `sharing.spec.js › KNOWN-BUG: page has multiple role="status" elements (accessibility)`

The confidence badge (`role="status"`) and the share toast (`role="status"`) coexist on the page. Multiple live regions with the same role can conflict for screen-reader announcements. **Fix:** Change the confidence badge `role` to `"img"` with an `aria-label`, keeping the toast as the only `role="status"`.

## Discovery — API key sent as header, not query param

**File:** `src/utils/gemini.js:103–104`
**Test:** `analysis-flow.spec.js › Gemini request hits the generateContent endpoint with correct URL`

SDK v0.21.0 sends the API key via the `x-goog-api-key` request **header**, not as a `?key=…` query parameter. This is the expected SDK behaviour and is not a bug, but it's worth noting for debugging network logs.
